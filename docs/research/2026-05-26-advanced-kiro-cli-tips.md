# Research: Advanced Kiro CLI Configuration — Matching and Exceeding Claude Code

## Summary

Kiro CLI is AWS's rebranded successor to Amazon Q Developer CLI, rebuilt around a spec-driven development philosophy and a layered configuration system (steering files, custom agents, hooks, MCP servers, and skills). It provides a structured, composable alternative to Claude Code's prompt-and-permission model. Its standout differentiators are: persistent context via markdown steering files with four inclusion modes, event-driven workflow hooks at the agent lifecycle level, a full custom-agent JSON system with per-agent model selection and tool sandboxing, parallel subagent orchestration (up to 4 concurrent), and a progressive-disclosure skills system that keeps context windows lean. Configured well, Kiro CLI matches Claude Code on interactive development and exceeds it on multi-agent orchestration, spec-driven planning, and workflow automation.

---

## Key Findings

### 1. Architecture Overview

Kiro CLI is the terminal-interface counterpart to the Kiro IDE (VS Code-compatible, built on Code OSS). Released publicly on July 14, 2025, rebranded from Amazon Q Developer CLI in November 2025. All persistent configuration lives under two roots:
- **Per-project**: `.kiro/` at the workspace root
- **Global**: `~/.kiro/` in the home directory

Workspace settings always take precedence over global settings.

---

### 2. Steering Files — Persistent Context (equivalent to `CLAUDE.md`)

Steering files are Kiro's equivalent of Claude Code's `CLAUDE.md` and `~/.claude/rules/`, but more structured with four inclusion modes.

**File locations:**
- `.kiro/steering/*.md` — project scope
- `~/.kiro/steering/*.md` — global scope

**Three foundational files** (generated by Kiro, always-included):
- `product.md` — purpose, users, features, business goals
- `tech.md` — stack, frameworks, constraints
- `structure.md` — file layout, naming conventions, architecture

**YAML frontmatter controls inclusion mode** (must be first content, no blank lines before):

```markdown
---
inclusion: always
---
# API REST Conventions
Always use plural resource names. Return RFC 7807 error bodies.
```

**Four inclusion modes:**

| Mode | Behavior | Use Case |
|------|----------|----------|
| `always` | Loaded into every interaction | Universal coding standards |
| `fileMatch` | Loads when working with matching file patterns | Stack-specific rules |
| `auto` | Loads when request semantically matches description | On-demand context |
| `manual` | Available via `#filename` in chat | Reference docs, rarely needed |

**FileMatch example** — only loads when editing React components:
```markdown
---
inclusion: fileMatch
fileMatch: "src/components/**/*.tsx"
---
# React Component Standards
Use functional components with named exports.
```

**Live file references** — embed live workspace content inside steering:
```markdown
#[[file:src/api/types.ts]]
```

**AGENTS.md compatibility:** Place `AGENTS.md` at project root or `~/.kiro/steering/` and it is always included automatically without frontmatter.

**Best practice:** One domain per file. Use descriptive names like `api-rest-conventions.md`, `testing-unit-patterns.md`, `security-secrets-policy.md`.

### 3. Custom Agent Configuration

Agents are JSON files stored in `.kiro/agents/` (project) or `~/.kiro/agents/` (global). The filename without `.json` is the agent's name. Local agents override global agents with the same name.

**Creation commands:**
```bash
kiro-cli agent create backend-specialist -D "Backend coding specialist"
kiro-cli agent create my-agent --manual
kiro-cli agent create my-agent --from backend-specialist
```

**Complete configuration reference:**

```json
{
  "name": "aws-rust-agent",
  "description": "Specialized for AWS infrastructure and Rust development",
  "welcomeMessage": "Ready to build AWS services in Rust.",
  "keyboardShortcut": "ctrl+a",
  "model": "claude-sonnet-4.6",
  "prompt": "file://./prompts/aws-rust-system-prompt.md",
  "tools": ["read", "write", "shell", "aws", "grep", "glob", "subagent", "@git"],
  "allowedTools": ["read", "glob", "grep", "@git/git_status", "@git/git_diff"],
  "toolAliases": {
    "@github-mcp/get_issues": "github_issues"
  },
  "toolsSettings": {
    "write": {
      "allowedPaths": ["src/**", "tests/**"],
      "deniedPaths": ["~/.ssh/**", ".env*"]
    },
    "shell": {
      "allowedCommands": ["cargo build", "cargo test", "git status"],
      "deniedCommands": ["rm -rf"]
    },
    "subagent": {
      "availableAgents": ["reviewer", "tester"],
      "trustedAgents": ["reviewer"]
    }
  },
  "resources": [
    "file://.kiro/steering/**/*.md",
    "file://README.md",
    "skill://.kiro/skills/**/SKILL.md"
  ],
  "mcpServers": {
    "git": {
      "command": "uvx",
      "args": ["mcp-server-git"],
      "timeout": 30000
    }
  },
  "includeMcpJson": true,
  "hooks": {
    "agentSpawn": [
      { "command": "git status && git log --oneline -5" }
    ],
    "preToolUse": [
      {
        "command": "bash .kiro/hooks/secret-scan.sh",
        "matcher": "write"
      }
    ]
  }
}
```

**Key fields:**
- `prompt` — accepts inline string or `file://` reference to a markdown file
- `tools` — capabilities the agent can use; `"*"` enables everything, `"@server_name"` includes all tools from an MCP server
- `allowedTools` — tools auto-approved without prompting
- `toolsSettings` — per-tool sandboxing: `allowedPaths`, `deniedPaths`, `allowedCommands`
- `includeMcpJson` — when `true`, inherits servers from global/workspace `mcp.json`

### 4. Model Selection Per Agent

| Model | Context | Cost Multiplier | Best For |
|-------|---------|-----------------|----------|
| Claude Opus 4.7 | 1M | 2.2x | Complex reasoning |
| Claude Sonnet 4.6 | 1M | 1.3x | Balanced coding |
| Claude Haiku 4.5 | 200K | 0.4x | Fast iteration / routing |
| Qwen3 Coder Next | 256K | 0.05x | Long agentic sessions |
| **Auto** | — | 1.0x | Intelligent routing (recommended default) |

**Per-agent model in JSON:**
```json
{ "model": "claude-sonnet-4.6" }
```

**Session-level:**
```
/model claude-haiku-4.5
/model set-current-as-default
```

**Power-user pattern:** Opus for orchestrators doing complex planning, Haiku for reviewer/router subagents, Qwen3 for long-running code generation.

### 5. Hooks — Workflow Automation

Hooks are event-driven automations that run shell commands at defined lifecycle points.

**Five event types:**

| Hook | When | Use Case |
|------|------|----------|
| `agentSpawn` | Agent activates | Inject git context, load env |
| `userPromptSubmit` | User sends message | Pre-validate, log prompts |
| `preToolUse` | Before any tool runs | Block dangerous ops, secret scan |
| `postToolUse` | After tool completes | Audit log, trigger downstream |
| `stop` | Assistant finishes | Generate changelogs, notify |

**Hook receives event via STDIN (JSON):**
```json
{
  "hook_event_name": "preToolUse",
  "cwd": "/Users/you/project",
  "tool_name": "fs_write",
  "tool_input": { "path": "src/auth.ts", "content": "..." }
}
```

**Exit code semantics:**
- `0` — success; STDOUT appended to agent context
- `2` — (`preToolUse` only) block tool execution, return STDERR to LLM
- other — failure; show warning, continue

**Secret scanning hook example:**
```json
{
  "hooks": {
    "preToolUse": [
      {
        "command": "bash ~/.kiro/hooks/secret-scan.sh",
        "matcher": "write",
        "timeout_ms": 10000
      }
    ]
  }
}
```

**Cache hooks for performance:**
```json
{
  "agentSpawn": [
    {
      "command": "cat package.json | jq .dependencies",
      "cache_ttl_seconds": 300
    }
  ]
}
```

### 6. MCP Server Integration

**File locations:**
- `.kiro/settings/mcp.json` — workspace
- `~/.kiro/settings/mcp.json` — global

**Local server:**
```json
{
  "mcpServers": {
    "git": {
      "command": "uvx",
      "args": ["mcp-server-git", "--repository", "."],
      "autoApprove": ["git_status", "git_diff", "git_log"],
      "disabledTools": ["git_push"]
    }
  }
}
```

**Remote server with auth:**
```json
{
  "mcpServers": {
    "github": {
      "url": "https://api.githubcopilot.com/mcp/",
      "headers": { "Authorization": "Bearer ${GITHUB_TOKEN}" },
      "autoApprove": ["*"],
      "disabledTools": ["delete_repository"]
    }
  }
}
```

**Key properties:**
- `autoApprove` — tools that run without prompting (supports `"*"` for all)
- `disabledTools` — tools hidden from the agent entirely
- `timeout` — per-request timeout in ms (default: 120,000)

**CLI management:**
```bash
kiro-cli mcp add --name git --command "uvx mcp-server-git"
kiro-cli mcp list
kiro-cli mcp remove --name git
```

### 7. Skills — Progressive Disclosure Tool Packages

Skills are Kiro's equivalent of reusable prompt libraries, with lazy loading to keep context lean.

**File layout:**
```
.kiro/skills/
  pr-review/
    SKILL.md
    references/
      checklist.md
```

**`SKILL.md` format:**
```markdown
---
name: pr-review
description: Security-focused pull request review checking for injection, secrets exposure, auth bypasses, and input validation gaps
---

## PR Review Process
1. Run `git diff main` to get the changeset
2. Check for secrets: API keys, tokens, hardcoded credentials
3. Verify input validation at trust boundaries

$ARGUMENTS accepts PR number or branch name.
```

**Activation:**
- Automatically when request semantically matches description
- Explicitly via `/pr-review 142` (passes `142` as `$ARGUMENTS`)

**Agent resource reference:**
```json
{
  "resources": ["skill://.kiro/skills/**/SKILL.md"]
}
```

### 8. Parallel Subagent Orchestration (Kiro's biggest advantage over Claude Code)

Up to 4 subagents run concurrently. Monitor with `Ctrl+G`, navigate with `Ctrl+D`/`Ctrl+U`.

**Orchestrator agent JSON:**
```json
{
  "name": "orchestrator",
  "model": "claude-opus-4.7",
  "tools": ["subagent", "read", "glob"],
  "toolsSettings": {
    "subagent": {
      "availableAgents": ["python-specialist", "reviewer", "docs-writer"],
      "trustedAgents": ["reviewer"]
    }
  },
  "prompt": "You coordinate development. Never write code directly. Break tasks into a DAG and delegate each node to the appropriate specialist. Use reviewers after every implementation."
}
```

**Explicit spawn:**
```
/spawn "Analyze the auth module for security issues" --agent reviewer
```

### 9. Prompt Engineering for Agents

1. **Externalize system prompt** to a markdown file:
   ```json
   { "prompt": "file://./prompts/specialist.md" }
   ```

2. **Structure the system prompt in layers:**
   ```markdown
   ## Role
   You are a Python specialist following TDD.

   ## Workflow
   1. Write tests first. Run them (expect failure).
   2. Write minimal code to pass. Run tests again.
   3. Refactor. Commit.

   ## Constraints
   - Never skip tests.
   - Never commit directly to main.

   ## Knowledge
   - Project uses pytest and Poetry.
   - Database layer is in src/db/, never mock it; use Docker test containers.
   ```

3. **Inject dynamic context via `agentSpawn` hooks** rather than hardcoding:
   ```json
   "hooks": {
     "agentSpawn": [
       { "command": "echo 'Current branch: ' && git branch --show-current && git log --oneline -3" }
     ]
   }
   ```

4. **Steering for shared constraints, agent `prompt` for role-specific behavior.**

### 10. Power-User Features Unique to Kiro CLI

| Command | Description |
|---------|-------------|
| `/checkpoint` | Snapshot filesystem state mid-conversation; restore if agent goes off-rails |
| `/compact` | Compress conversation history, extending effective context window |
| `/rewind` | Fork conversation at an earlier point |
| `/knowledge` | Semantic search across project files, persisted across sessions |
| `/effort` | Set reasoning effort level per session |
| `/translate` | Convert natural language to shell commands |
| `/spawn` | Explicitly spawn a named subagent |

**Spec-driven development** (unique to Kiro): `.kiro/specs/<feature>/` folders contain `requirements.md`, `design.md`, and `tasks.md`. Tasks execute with dependency-aware parallelism (wave execution).

### 11. Mapping Claude Code → Kiro CLI

| Claude Code | Kiro CLI Equivalent |
|-------------|---------------------|
| `CLAUDE.md` / `~/.claude/rules/` | Steering files (`.kiro/steering/*.md`) with inclusion modes |
| `settings.json` permissions | Agent `toolsSettings` + `allowedTools` |
| `/project:{name}` slash commands | Skills (`/skill-name`) + manual steering (`#filename`) |
| Single session, single model | Per-agent `model` field + `/model` in-session |
| `--trust-all-tools` | `kiro-cli chat --trust-all-tools` |
| MCP via config JSON | `.kiro/settings/mcp.json` with `autoApprove` and `disabledTools` |
| No native multi-agent | `subagent` tool, up to 4 parallel, DAG orchestration |
| No built-in workflow triggers | 5 hook event types across agent lifecycle |

## Trade-offs / Caveats

- **IDE vs. CLI parity**: File-save hooks are IDE-only. CLI hooks cover agent lifecycle events only.
- **DAG task graphs are immutable**: Task graphs are planned upfront and cannot be modified mid-execution — restart with a revised plan if a task fails.
- **`autoApprove: ["*"]` is a security risk**: Auto-approves all tools including destructive ones. Use explicit tool names in production.
- **Model availability varies by region**: Kiro routes through AWS Bedrock; not all models are available in all regions.
- **Global `always` steering can bloat context**: Reserve `always` for truly universal rules; prefer `fileMatch` or `auto` for domain-specific content.
- **`inclusion: auto` has known reliability bugs**: GitHub issue #7787 — verify it works in your Kiro version before relying on it.
