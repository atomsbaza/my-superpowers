# Chapter 24: Acceptance Testing

## Core Idea
Natural-language requirements are inherently ambiguous, and the only reliable way professionals have found to eliminate that ambiguity is to write automated acceptance tests, jointly owned by business and developers, that formally define what "done" means.

## Frameworks Introduced
- **Acceptance Tests as the Definition of Done**: "Done means all code written, all tests pass, QA and the stakeholders have accepted." Acceptance tests are executable specifications, written by a collaboration of stakeholders and programmers, that define precisely when a requirement is complete.
  - When to use: For every requirement/story before implementation begins — ideally written a few days ahead of the developer starting the work (the "late precision" principle).
  - How: Stakeholders (or delegates: business analysts, QA) express the requirement as concrete given/when/then style scenarios in a business-readable, executable format (e.g., FitNesse tables, Cucumber-style Gherkin). Developers then connect these tests to the system and implement until they pass. Business analysts typically write the "happy path" tests; QA writes the "unhappy path," boundary, and exception tests.

## Key Concepts
- **Acceptance test**: A test written jointly by stakeholders and programmers that defines when a requirement is done — distinct from tests users run before accepting a release or from QA-only tests.
- **Premature precision**: The trap of both business and developers wanting exact specifications too early, before contact with a working system reveals what's actually needed.
- **The uncertainty principle (observer effect)**: Demonstrating a working feature gives stakeholders new information that changes what they think they want — so precision fixed too early becomes irrelevant once the system is built.
- **Late ambiguity**: Deferring precision (correctly, to fight premature precision) can let unresolved stakeholder disagreements hide inside vaguely-worded requirements instead of being resolved.
- **Contextual ambiguity**: A requirement can be perfectly clear to the speaker in their own context yet mean something entirely different to the listener, even in direct face-to-face conversation.
- **Done vs. done-done**: The dangerous ambiguity of teams using informal, undefined gradations of "complete" instead of one rigorous definition.
- **Requirements as executable documents**: Acceptance and unit tests are primarily formal documents that specify design/behavior; that they also verify automatically is a (valuable) side effect, not their main purpose.
- **Testing through the API, not the GUI**: Acceptance tests should invoke business rules through a stable API layer beneath the GUI, since GUIs are aesthetically volatile and tests coupled to GUI layout are fragile.
- **Continuous integration as an enforcement mechanism**: All unit and acceptance tests run several times a day in CI; a broken build is a "stop the presses" emergency, not a backlog item.

## Mental Models
- **The sculptor and the tool**: Requirements are discovered, not specified — the stakeholder is the sculptor shaping the system through iteration, and early contact with a working prototype (or acceptance test) is what actually reveals the real requirement, far better than upfront documents.
- **An ambiguity is a buried argument**: Quoting Tom DeMarco, an ambiguous phrase in a requirements document usually represents an unresolved disagreement among stakeholders that wordsmithing papered over rather than settled.
- **Tests as formal documents, not extra work**: Acceptance tests are not additional QA overhead layered on top of "real" specification — they *are* the specification, just precise enough to execute. Viewing them as "extra work" is a category error; the alternative (manual test plans, ambiguous prose) is strictly more expensive.
- **Manual testing as a sunk-cost disaster**: The anecdote of the QA manager forced to cut a million-dollar, six-week manual test cycle in half — literally choosing which half of the product to stop verifying — reframes automation not as a nice-to-have but as the only economically sane option.

## Anti-patterns
- **Premature precision**: Both business (wanting exact deliverables before authorizing) and developers (wanting exact specs before estimating) chase a precision that cannot be achieved before the system exists, wasting effort refining details that will change once stakeholders see the running system.
- **Ambiguous natural-language requirements left unresolved**: Verbal or prose requirements (the log-backup dialogue between Sam, Paula, and Peter) let each party silently substitute their own interpretation ("save the logs" meant "keep one backup" to Paula, "keep every log forever" to the customer) — nothing forces the gap to surface until production data is lost.
- **"Done" left undefined until QA finds gaps**: Without an explicit, testable definition of done, "done" silently degrades to "I wrote code and ran it once," pushing discovery of missing behavior downstream to QA or, worse, to customers.
- **Manual acceptance/regression test plans**: Manual execution is so costly (person-hours, calendar time, off-shore test armies) that teams eventually can't afford to run their own test plan and start skipping coverage — the QA manager's "which half of these tests should I not run?" moment.
- **Coupling acceptance tests to GUI structure**: Selecting elements by screen position or fragile DOM structure instead of stable IDs/APIs makes tests break on every cosmetic GUI change, causing teams to either delete the tests or freeze the UI — both bad outcomes.
- **Passive-aggressive test compliance**: A developer who finds a test wrong, awkward, or based on a bad assumption but implements it literally anyway ("well, that's what the test says") instead of negotiating a better test with its author, quietly encoding a known defect into "passing" software.
- **Treating unit tests and acceptance tests as redundant**: They exercise the same behavior through different pathways (unit tests hit internal classes/methods; acceptance tests hit the system at the API/UI level) and serve different audiences (programmers vs. business) — dropping one to save "duplicate" effort loses real coverage and communication value.
- **Letting CI stay broken**: Setting aside failing tests "to fix later," or even removing them from the build under time pressure, silently regresses coverage; the chapter's cautionary case is a team that forgot to restore removed tests and only found out via angry customer bug reports.

## Worked Example
Martin builds the case for automated, jointly-owned acceptance tests through the extended Sam/Paula/Tom log-backup scenario. Initially Sam states the requirement in prose ("these log files need to be backed up daily"); Paula, a developer, extracts what seem like sufficient details (directory name "backup," time "midnight") through Q&A, but the conversation still hides a critical ambiguity — Sam's customer Carl expects *every* log file ever produced to be retained, while Paula understood only last night's log needed saving. Nothing in the prose-based conversation surfaces this gap; it would only appear in production when historical logs turned out to be missing.

The chapter then replays the same conversation with Tom, a tester, added to the discussion. Tom's insistence on precise, testable language ("backup" is renamed to `old_inactive_logs` to force clarity about what's actually stored, and "many logs, not one" surfaces immediately) exposes the ambiguity the moment it is discussed, because the goal of the conversation is now to produce a concrete, executable test rather than just to reach verbal agreement. Tom writes the acceptance criteria as literal given/when/then scenarios against a `LogFileDirectoryStartupCommand`, expressed in a FitNesse-style table so they can run against the code directly rather than the whole system (avoiding the 20-second, growing system startup cost Paula flags).

This motivates the chapter's central argument: writing acceptance tests is not "extra work" layered on top of real specification — it *is* the act of specifying the system precisely enough that ambiguity cannot survive, and the byproduct is a test suite that automatically tells everyone when the feature is done. The developer's role, illustrated by Peter picking up Paula's failing acceptance test and incrementally making scenarios pass (using shared, reusable Fixture/Scenario plumbing), shows how implementation work should start only once such tests exist and initially fail — implementation is "just" the work of turning red tests green. The later Z-score negotiation between Paula and Tom over a "finishes within 2 seconds" performance requirement demonstrates the "test negotiation" principle: when a test as written doesn't reflect achievable reality (a hard 2-second guarantee is impossible; a 99.5% statistical guarantee is what's actually deliverable), professionals renegotiate the test itself rather than silently building to a test they know is wrong.

## Key Takeaways
1. Never accept ambiguous, prose-only requirements as a definition of done — force them into concrete, executable acceptance tests before implementation starts.
2. Defer precision as long as practical (late precision) to avoid premature-precision waste, but write full acceptance tests before development on a feature begins, closing the "late ambiguity" gap stakeholder disagreement can hide in.
3. Acceptance tests must be automated; the economics of manual test execution eventually force teams to cut corners and fly blind on parts of their own product.
4. Ownership is shared: business analysts typically own happy-path tests, QA typically owns unhappy-path/boundary/exception tests, and developers implement against and can negotiate the tests — but a developer should never implement tests they personally wrote for their own feature.
5. When a test seems wrong, awkward, or unachievable, negotiate a better test with its author instead of passively implementing to a known-bad specification.
6. Design systems, especially GUIs, so acceptance tests can exercise behavior through a stable API layer rather than fragile UI structure, keeping tests durable as presentation details change.
7. Treat a broken CI build as a "stop the presses" emergency — the acceptance-test suite is only trustworthy as a definition of done if it is never allowed to stay red.
