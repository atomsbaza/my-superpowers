# TDD Knowledge Base — Glossary

**Acceptance Tests / ATDD** — tests verifying the system meets business/user criteria, defined via the Three Amigos before coding; sits atop the testing pyramid (Topic 5, Topic 6)
**Big Ball of Mud** — tangled, poorly structured code that a strong test suite makes safe to refactor (Topic 13)
**BDD (Behavior-Driven Development)** — expresses system behavior in Given/When/Then (Gherkin) so non-programmers can read and validate specs (Topic 6)
**Bowling Game Kata** — Uncle Bob's canonical practice kata for drilling TDD discipline (Topic 11)
**Canon TDD** — Kent Beck's formulation of red-green-refactor plus supporting techniques: Test List and Green Bar Patterns (Topic 2)
**Characterization Testing** — tests that lock in what legacy code actually does (not what it should do), as a safety net before refactoring (Topic 10)
**Coding Kata** — small, repeatable exercise (e.g., Bowling Game, FizzBuzz) for deliberate TDD practice (Topic 11)
**Contract Tests** — tests validating the communication boundary between two systems/services (Topic 5)
**Detroit School (Classicist)** — state verification, minimal mocking, module-level units; originated with Kent Beck (Topic 1, Topic 3)
**Excessive Setup** — anti-pattern: huge arrange/setup blocks that obscure test intent and are costly to maintain (Topic 8)
**Fake It ('Til You Make It)** — returning a hardcoded value to pass a test quickly, deferred to real logic later (Topic 2)
**Gherkin** — Given/When/Then DSL for describing behavior in plain English (Topic 6)
**Golden Master Technique** — bulk output-comparison characterization testing for tangled/opaque legacy systems (Topic 10)
**Green Bar Patterns** — Fake It, Obvious Implementation, Triangulation: Beck's three strategies for reaching Green (Topic 2)
**Inside-Out (Bottom-Up)** — design flow from core logic outward to UI; pairs with Detroit/state verification (Topic 4)
**Interaction Verification** — asserting on how the SUT called its collaborators; characteristic of the London school (Topic 3)
**London School (Mockist)** — interaction verification, extensive mocking, class-level units; formalized by Freeman & Pryce's GOOS (Topic 1, Topic 3)
**Living Documentation** — executable BDD scenarios that stay accurate because they fail on behavioral drift (Topic 6)
**Mocking What You Don't Own** — anti-pattern: mocking third-party code directly instead of via an owned adapter (Topic 3, Topic 8)
**Obvious Implementation** — writing the real, correct implementation directly when it's simple and known (Topic 2)
**Outside-In (Top-Down)** — design flow from a user scenario inward to logic; pairs with London/interaction verification (Topic 4)
**Red-Green-Refactor Cycle** — the core TDD loop: write a failing test, make it pass minimally, then clean up (Topic 2)
**Self-Testing Code** — Fowler's goal (a suite giving confidence to change code); TDD is one route to it, not the destination (Topic 12)
**Spy** — test double that records calls for later inspection (Topic 3)
**State Verification** — asserting on final observable state after an operation; characteristic of the Detroit school (Topic 3)
**Stub** — test double returning hardcoded answers to calls made during a test (Topic 3)
**Subject/System Under Test (SUT)** — the specific class or method a test verifies; narrowed to one class per test in the London school (Topic 3)
**Test-Induced Design Damage** — DHH's term for architectural harm from indirection added purely for testability (Topic 8, Topic 12)
**Test Data Builder** — Builder-pattern helper for constructing readable test fixtures with sensible defaults (Topic 13)
**Test Double** — umbrella term for any stand-in object replacing a real dependency (Mock, Stub, Fake, Spy) (Topic 3)
**Test List** — a running backlog of test scenarios/variants maintained alongside code, part of Canon TDD (Topic 2)
**Testing Pyramid** — layered test-suite model: many fast unit tests at the base, fewer, slower tests above (Unit, Integration, Contract, Acceptance, E2E) (Topic 5)
**The Inspector** — anti-pattern: assertions reaching into internal state, brittle to safe refactors (Topic 3, Topic 8)
**The Liar** — anti-pattern: a test that passes without truly verifying the requirement (Topic 8)
**The Mockery** — anti-pattern: over-mocked tests that only confirm mock interactions, not real behavior (Topic 3, Topic 8)
**The Slow Poke** — anti-pattern: tests too slow to run often, eroding the fast-feedback loop (Topic 5, Topic 8)
**Three Amigos** — Business, Developer, and Tester collaborating on acceptance criteria before coding (Topic 4, Topic 5, Topic 6)
**Transformation Priority Premise (TPP)** — Robert C. Martin's prioritized list of code transformations, simplest first, when making a test pass (Topic 2, Topic 9)
**Triangulation** — generalizing code only once two or more specific tests jointly demand it (Topic 2)
**Unit Tests** — smallest-scope tests exercising a single unit in isolation; the pyramid's largest, fastest layer (Topic 5)
**Vulnerable Tests** — tests (typically London-style) that break on internal refactors despite unchanged output (Topic 3, Topic 8)
**XP's Five Core Values (in testing)** — Communication, Simplicity, Feedback, Courage, Respect, as expressed through test code quality (Topic 13)
